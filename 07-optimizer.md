# Optimizer

## Purpose

The Optimizer is the last per-channel component before the Comparator. It receives a plan that has already passed all validity checks and transforms it into canonical form: a standard, unambiguous representation in which semantically equivalent plans produce the same JSON structure.

The Optimizer exists directly because of the Comparator: two equivalent plans represented differently would be seen as different plans during comparison. This would produce false disagreements that degrade system availability without corresponding to real errors.

The Optimizer resolves this before it reaches the Comparator, applying rewriting rules that collapse equivalent variants into a single form.

---

## The Capability Registry as primary input

The Optimizer does not apply generic rewriting rules: its decisions are guided by the Capability Registry. For every contextual node in the plan, the Optimizer consults the `supports` section of the registry for that resource and action, and determines which subsequent transversal nodes in the linked list can be collapsed.

This means the Optimizer has no domain logic hardcoded internally: all knowledge about what is collapsible lives in the registry. Adding native support for a new filter field on an external API requires only updating the registry — the Optimizer handles it automatically.

The separation also matters in the negative direction: if a field is not declared as supported in the registry, the Optimizer does not collapse it, even if the API might technically accept it. A declaration in the registry is the necessary and sufficient precondition.

---

## Collapsing transversal nodes

The Optimizer's primary transformation is the collapsing of transversal nodes into the preceding contextual node. The plan produced by the Planner is intentionally explicit and unoptimized: every filter, sort, and limit is a separate node in the linked list. The Optimizer walks the list and, for each contextual node, checks in the registry which of the subsequent transversal nodes can be absorbed.

A `filter` node can be collapsed if the field it operates on is declared in `supports.filter.fields` of the preceding contextual node, and if the operator used is among those listed for that field. If both conditions are satisfied, the filter node is removed from the list and its parameters are added to the parameters of the contextual node, which will transmit them to the external API in the HTTP call.

A `sort` node can be collapsed if the field is listed in `supports.sort.fields`. A `limit` node can be collapsed if `supports.limit` is not `false`.

Transversal nodes that do not satisfy the conditions remain in the list as client-side nodes: they will be executed locally after the contextual node has received the API response.

---

## Canonicalization as the objective

The formal objective of the Optimizer is to create a function that maps every equivalence class of plans to a single representative. Two plans that produce the same result when executed must produce the same output from the Optimizer.

This is an approximation objective, not a completeness objective: it is neither necessary nor generally possible to capture all possible equivalences. It is sufficient to capture the equivalences that appear frequently in plans generated by the models, thereby reducing the false disagreement rate to an acceptable level.

---

## Rewriting rules

The Optimizer's rewriting rules are local graph transformations that preserve semantics. Each rule has an explicit precondition that delimits its applicability.

**Filter commutativity** is the canonical example: two sequential filter nodes with no dependencies between them produce the same result regardless of order. The Optimizer can sort them canonically — for example, alphabetically by the filtered field. The precondition is that the two filters are genuinely independent: if the second filter uses a field computed by the first, commutativity does not hold and the rule does not apply.

**Collapsing sequential filters**: two sequential filter nodes can be collapsed into a single filter node with conditions combined via AND. The precondition is that the filters are independent and that the downstream execution engine supports composite conditions. If this assumption does not hold, the rule must not be included.

**Node identifier normalization**: if plan nodes are named with identifiers that vary between generations (auto-generated names, UUIDs, etc.), the Optimizer can rename them canonically — guaranteeing that all references in the dependency list are also updated consistently.

**Dependency ordering normalization**: if the dependency lists of each node have no defined semantic ordering, the Optimizer sorts them canonically (for example, by the identifier of the dependent node).

---

## The soundness principle

The most important property of the Optimizer is soundness: it must never produce a plan that behaves differently from the original when executed. It may be incomplete (not finding all equivalences), but it must not be incorrect (not modifying semantics).

This principle requires that every rewriting rule be accompanied by a demonstration — even informal but explicit — that the transformation preserves semantics, and by tests that verify this property on concrete cases.

The temptation to add "obvious" optimization rules without rigorously verifying their preconditions is the main risk of this component.

---

## The risk of lossy optimizations

A lossy optimization is a transformation that changes the plan's behavior in a way that is undetected by the validators. This is the Optimizer's most insidious failure mode because it operates after all validity checks.

The typical case is a rule that assumes the commutativity or idempotence of an operation that does not hold it in all circumstances. A filter that executes a query on an external system as a side effect is not idempotent: collapsing it with another filter changes the plan's behavior in a way that is not visible from the structure.

The primary defense is explicit documentation of each rule's preconditions and testing on edge cases. It is not sufficient for the rule to be correct in the common case — it must also be verified at boundary conditions.

---

## Practical example

A plan produced by channel A has these nodes in sequence: fetch, filter by status, filter by amount, sort, limit. Channel B produced: fetch, filter by amount, filter by status, sort, limit. Both are semantically equivalent: the two filters operate on different fields and can be applied in either order.

Without the Optimizer, the Comparator would see a topological difference between the two plans and signal a disagreement. With the Optimizer, both plans are transformed into: fetch, filter by amount AND status (collapsed into one node), sort, limit. The Comparator sees two identical plans and signals agreement.

This is the ideal case. A more complex case emerges when the two filters are not completely independent — for example, if the amount filter must be applied after the status filter has already narrowed the set, for efficiency reasons or due to constraints of the execution engine. In this case the collapse rule does not apply and the filter order must be preserved.

---

## The relationship with the Comparator

The Optimizer and the Comparator are designed in tandem. The Comparator defines the level of detail at which comparison occurs; the Optimizer ensures that differences irrelevant at that level have been eliminated before comparison.

If the Comparator compares at the topological level — same dependency graph, same node types, same structure — the Optimizer must guarantee that topologically equivalent variants produce the same topological structure.

---

## Advantages of this approach

The Optimizer improves system availability by reducing false disagreements at the Comparator. Without it, trivial style variations in plan generation — filter order, identifier names — would produce disagreements that do not correspond to real semantic differences.

The Optimizer is also the only component that can simplify the plan, reducing the number of nodes where equivalent nodes exist. A simpler plan is easier to execute, debug, and understand.

---

## Drawbacks, open questions, and known issues

The Optimizer is the component with the greatest risk of introducing errors silently. It operates after all validators, so a lossy optimization is not detected by any subsequent check in the pipeline. The responsibility for guaranteeing the correctness of rewriting rules falls entirely on the people who design and maintain the component.

The fact that the Optimizer is identical on both channels means that a bug in it manifests equally on both, producing agreement at the Comparator even when both plans have been transformed incorrectly.

An open question concerns the completeness of canonicalization. How many equivalences exist in the specific domain? This is difficult to answer in advance. In practice, equivalences are discovered by observing false disagreements at the Comparator over time: every disagreement classified as an "equivalent variant" is a candidate for a new Optimizer rule.

This suggests an iterative process: start with a few well-verified rules, observe false disagreements in production, add rules progressively. This is preferable to starting with many rules and discovering only in production that some were wrong.
